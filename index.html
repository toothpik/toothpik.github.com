things in mind, go to the vim homepage.

This page is more about how to have fun with vim than anything, or at least how I have fun with it.

It's also about how to have fun with python and tcl and perl

...and bash, awk, grep, sed, and all the rest.

Lately I am inclined to build an interpreter free vim - no python, tcl, or perl linked in - mainly
this is a tactic in the "quicker-enter-to-running" war - it is also backed with the understanding
that if you want to do something to your buffer in a language other than VimL you can write
a function to save your buffer, then call an external script with system(), or :r!<scriptname>,
or :[range]!<scriptname>, or any of a dozen other ways.

konsole four
konsole four is for budgeting - here I track my checking account, reconcile, and project it into the
future - over the years I've used a variety of spreadsheets to store this data, and have repeatedly
been frustrated by upgrades to the software making my older data difficult to get to - the answer
I've come to embrace is ascii text - I cannot imagine a time when text is unreadable - ok, yes, I can
if I stray into disaster science fiction, but if we get to that point we won't have banks and checking
accounts anyway - whether the scripting language that reads it evolves is irrelevant - the data is
the important thing, and text is the most universally readable format available

by formatting the data in specific columns I enable a variety of languages easy access to it - the
FIELDWIDTHS feature of awk springs to mind, but every language I know can parse lines of a known
format

copying my last spreadsheet I defined the format for what I refer to as the plan modules as

    column 1 - cc1-5 - date of transaction, format m/d (right justified)
    column 2 - cc7-9 - type of transaction
    column 3 - cc11-40 - description (name of merchant)
    column 4 - cc42-43 - requested clear date for online transactions
    column 5 - cc45-46 - actual clear date
    column 6 - cc48-57 - transaction amount
    column 7 - cc60 - blank for future, 'x' for past
    column 8 - cc62-70 - balance
    column 9 - cc72-80 - reconciliation amount
    column 10 - cc82-86 - reconciliation date

I have defined fiscal months as starting on the 23rd of the preceding month and ending on the
22nd - that way there's a full week of lead time for transactions targeted for the 1st, and the
month starts off clean with the first transactions first - that's my take on it, it's my system,
so I get to decide

the script I use the most for displaying it is m1, which uses curpmo to determine the month
to be displayed by m12

m uses a variable combination of mYY modules to display the relevant portion of my account, for
example:
    { m11 8; m11 9 } | head -31
will display august and most of september for 2011 - m starts by running the entire year (m11)
through funclear.awk in order to determine whether the first unclear transaction occurs in the
same or the previous fiscal month - with this information it can design the most relevant display
using two of
- plan1.sed - plan2.sed - plan3.sed - plan4.sed - plan5.sed - plan6.sed
- plan7.sed - plan8.sed - plan9.sed - plan10.sed - plan11.sed - plan12.sed

the plan_yyyy data modules have a very specific format, and I edit them by invoking gp, which
sources plan.vim as it calls up the module - any of planfood, planbal, or planrecon may be
invoked by different <Leader> mappings defined in plan.vim

m's brother m2 displays information from the current plan, like m, but m2 restricts itself to
unclear transactions - first m2 runs uclro, which cats the current plan through uclearo.awk
for the purpose of finding and displaying transactions with an 'x' in the past/future field but
no date in the clear-date field - with its FIELDWIDTHS feature awk is well suited for working
with data of this nature - m2 then pipes the uclro output through sumlwq.awk in order to
sum the transaction amounts of all unclear transactions - combining lastx, lastx.awk,
balamt.awk, solve (my favorite commandline calculator), and ciaa.awk m2 is able to display
the sum of unclear trans, the balance amount for the last transaction with an 'x', and the
reconciliation amount - these display nicely under an m display, and serve to tell me if
there might be transaction clear dates on my bank's web site

when it's time, I make the next year's plan using plannew, a bash script that combines a
tcl core, two python scripts, and five perl scripts, most of which are tailored to my specific
needs, with the exception of plandsh which puts a line of dashes between months at the
end of the series - driving all this is the plan_core which is described in the plan_readme -
of note is the last line in the core with description "food & atm" - the more accurate you
can be in estimating this important item the closer your future projections will be to reality -
a powerful tool in the quest for accurate food/atm estimates is a collaberation between lfy
(last fiscal year) and mft (make food totals) - as
    lfy | mft
a recent enhancement to lfy is cfm (current fiscal month) which makes use of cfm.awk
and perl/cfm

another tool that can be used to aid in estimating gas, electric, or other seasonal bills is
to pipe lfy into ghs, then in gvim locate the merchant of interest with '/' or '*' or even
:let @/ = 'merchant of interest', then hit ',o' - I currently use comma as my Leader for
mappings, and <Leader>o is mapped to a powerful fold expression that folds away
everything in the current buffer except lines containing whatever's in '@/'

konsole seventeen
cd7 (~/.build/vim) - here I start with dp, my "dangerous path" function defined in .bashrc
that adds $(pwd) to the beginning of $PATH, because all my vim build scripts are here -
next I may just do an hlsb2 and move on, or, if bram's been posting patches in vim_dev
I will start an update/build process without even checking hlsb

the first script run by hlsb2 is hlsb, which runs both hlslt and hlsld, making use of ee
along the way - the purpose of all this is to run t, which at its core performs an
'hg incoming' against the vim repository - all the 'hls_' modules test "how long since"
the last empty 't' or the last data hit - I don't want to unduly stress the server, especially
when there's no cause to believe any patches have been uploaded, so my rule is to not
't' any more often than once every 10 hours

rounding out the hlsb2 display is dbu, which displays the last 11 data hit dates, each
with a count of days since the one before, finishing with an average "days between
updates" calculation

when it all comes together it looks like this

if there are patches, I start xterm& from konsole 17 so the $(pwd) and $PATH will be
set for me already - in that xterm I run tfu, which starts a 'tail -f' on my update.log - I
arrange the xterm so I can watch it when I alt-tab back to konsole 17, where I start the
update process

there are three main scripts to update: u, u2, and u3 - u2 makes use of cs and lft.awk
to determine what to merge
